核心概念 — 工程师必读
SIS
Skin Improvement Score · 皮肤改善分数

SIS 是 DrRuby 的核心输出指标。它不是一个绝对的"皮肤好坏"评分，而是一个相对改善量：用同一个用户的干预侧（涂产品）与对照侧（仅基础方案：洁面 + 保湿 + SPF）的皮肤特征差值计算得出，抵消所有系统性混淆因素（月经周期、季节、压力等）。

SIS_delta = SIS（干预侧） − SIS（对照侧）
正值 = 干预侧比对照侧改善更多（产品有效）
零值 = 两侧无差异（产品无效）
负值 = 干预侧比对照侧更差（产品可能有刺激）

SIS 由 Haut.AI 从手机拍摄图像提取的皮肤特征计算：色素均匀度（pigmentation_index）、红斑密度（erythema_score）、毛孔评分（pore_density）、纹理均匀度（texture_uniformity）。权重来自皮肤科顾问判断 → 结构化转换流程（见下方），不由工程师自行决定。

⚠ 无超声设备，不存在 device_captures 表。深层皮肤生物物理数据（弹性、胶原密度）仅来自诊所访问（clinic_records），每?个月一次。Gen 1 居家数据源仅为：手机图像 → Haut.AI 特征。

SIS 权重来源

专家判断 → 公式：两阶段推导

Brenner 提供生物科学判断，皮肤科医生提供临床判断，Gunter 提供激素调整判断。这些判断通过结构化流程转化为可计算的公式——专家不需要懂统计，只需要看图评分或排序。

Gen 1
专家直接给出相对重要性
流程：向 Brenner + 皮肤科医生提问：
"色素均匀度、红斑密度、毛孔大小、纹理均匀度，对于皮肤改善的判断，哪个最重要？请排序并给出各自占比。"

例如专家给出：色素 ?% / 红斑 ?% / 纹理 ?% / 毛孔 ?%
这就是初始权重，直接用于 SIS 计算。（具体数值待 Brenner + 皮肤科医生会议后填入）
优点：无需数据，可立即启动工程实现
局限：权重主观，需在 Gen 2 用真实数据验证
Gen 2
真实数据回归推导权重
流程：积累 ?–? 组图像对（基线 vs 跟踪），让 Brenner + 皮肤科医生独立打分（0–10，改善了多少）。（样本量待统计顾问确认）跑线性回归：

SIS = w1×色素 + w2×红斑 + w3×毛孔 + w4×纹理 回归系数即为权重。专家只需看图打分，不需要懂数学。
优点：权重由真实临床判断数据推导，科学可辩护
前提：需要 Gen 1 积累足够标注图像
Gunter 妇科判断 — 进入解释层，不进入权重
Gunter 的激素判断不是 SIS 计算权重，而是 SIS 结果的调整系数和解读规则，存入专家知识库供 RAG 调用：
• "黄体期（排卵后至月经前），皮脂分泌增加、皮肤屏障功能可能下降，SIS 出现小幅波动属正常生理变化，不代表产品失效" → insight_feed 解释文字（具体波动范围需 Gunter 确认后填入）
• "绝经后雌激素下降，胶原流失加速，SIS 基线期望值应相应调低" → users.hormonal_status 驱动的解释逻辑
• "口服避孕药可能改变皮脂水平，confounder_log.medication 记录后需纳入分析" → 混淆因素控制
工程启动前必须完成： Ruby 组织 Brenner + 皮肤科医生完成 Gen 1 权重排序（会议时长待定）。产出物：四个特征的百分比权重，由 Brenner 签字确认。工程师收到权重后方可实现 SIS 计算模块。此项是工程关键路径上的依赖，不可跳过。

Vehicle-Controlled Design（皮肤科临床标准）
对照侧不是"什么都不涂"，而是涂标准化基础方案（指定洁面 + 基础保湿 + SPF50），干预侧在基础方案之上叠加测试产品。SIS_delta 测量的是测试产品相对于基础方案的增量效果，这是皮肤科 split-face 试验的标准设计，科学上可辩护，现实上可执行。

Washout 期：研究注册后，W0 采集前需完成 2周 washout（两侧统一只用指定洁面 + 基础保湿，停用所有其他产品），让皮肤状态归零再建立基线。washout_status 存入 study_sessions。

禁止清单（用户注册时签字确认，写入 consent_at）：26周内不做激光 / 医美 / 果酸换肤；不在未记录情况下更换产品；对照侧只涂基础方案，不额外添加任何产品。

W0 基线对称性检验：W0 采集完成后，后端自动计算 |SIS(干预侧) − SIS(对照侧)|，若超过阈值（待定）则写入 study_sessions.baseline_asymmetry_flag = true，不阻止研究，后续分析时作为协变量标注。

采集层

客户端 · 手机 App
  • 采集频率 — 每周2次（如周一 + 周四），间隔 ≥ 3 天；不可每日采集（过度采集影响依从性且无增益），不可每周1次（信号稀疏）
  • 采集时间引导 — 注册时询问"你通常什么时候拍比较习惯？"（早 / 中 / 晚），答案存入 study_sessions.preferred_capture_window，用于 App 推送提醒时间；不强制锁定，用户随时可拍。每次采集的 captured_at 时间戳完整记录，后端分析时作为协变量处理。理由：split-face 设计同次采集左右脸同时拍，时间段影响两侧相同，SIS_delta 相减已抵消，科学有效性不受损
  • AR 距离引导 — App 屏幕 AR 引导框帮助用户将手机对准面部（颧骨/眼下三角区），固定距离约 20cm，确保每次角度一致。⚠ 具体距离/角度标准待皮肤科顾问确认
  • 协议锁定 — 采集前皮肤状态清单强制确认（卸妆、洁面后再采集）
  • 环境光检测 — 采集前实时检测环境亮度（lux）；光线不足时强制弹出提醒，低于阈值不允许提交图像；lighting_score 写入 images 表。推荐在自然光或标准室内光下采集，不可逆光或强闪光灯。
  • 实时质量检测 — 清晰度、对焦、运动模糊、光线亮度四项，任一不通过则提示重拍，不记录本次图像
  • MCS 评分 — High / Medium / Low 置信度，采集时即时标记（光线是 MCS 评分的显式权重之一）
  • Context 录入 — 今日产品、月经周期天数（绝经用户跳过）、睡眠质量
  • 左/右脸标记 — 每张图像必须记录 face_side

⚠ 断层A修复 — 各 milestone_type 写入触发条件：
① first_positive_sis：每周 SIS 计算完成后，检查当前用户该 session 是否已有此类型记录；若 sis_delta > 0 且无历史记录 → INSERT 一条（只插一次，不重复）。
② sustained_?wk（周数见待确认问题 #11）：每周计算完成后，统计连续正向周数；达到阈值且无历史记录 → INSERT（只插一次）。
③ best_sis_ever：每周 SIS 计算后，与 health_journey.best_sis_score（需新增此字段）比较；若超出历史最高 → INSERT 并更新 health_journey.best_sis_score（可重复出现，每次创新高都插）。
④ clinic_elastometry_validated：clinic_records 写入触发（非每周 SIS 任务）——诊所上传弹性测量数据时，后端检查当前 session 是否已有此 milestone；没有则 INSERT。
⑤ sleep_improved：每周 SIS 计算完成后，检查同期 wearable_daily 睡眠均值是否高于用户基线 sleep_baseline（存于 health_journey）；是则 INSERT（同一 session 内只插一次）。

insight_feed 【数据反馈】系统自动发现的相关性洞察 — 每周后端跑相关性分析后推给用户 insight_id (主键，供 insight_rag_log FK 引用), user_id, session_id (相关性类洞察不可为 NULL；milestone_approaching / cohort_recommendation 可为 NULL), insight_date, insight_type (skin_sleep_correlation / sis_drop_explained / lifestyle_skin_link / milestone_approaching / cohort_recommendation), metric_a, metric_b, correlation_r, message_zh, delivered_at (nullable)。⚠ session_id 约束：skin_sleep_correlation / lifestyle_skin_link / sis_drop_explained 三类洞察必须关联到具体 session，否则无法确定是哪轮研究的 SIS_delta；工程师在写入时必须校验，session_id = NULL 时拒绝写入这三类 insight_type。 每周 SIS 计算完成后自动生成；推送条件：correlation_r ≥ ? 且 n ≥ ? 周有效数据（⚠ 阈值待统计顾问确认：n 太小时 r 值无统计显著性，会发出误导性洞察；建议 Brenner 或统计顾问给出最小样本量和 r 下限，确保洞察科学可信）。Gen 1 分析方法：Spearman 秩相关（SIS_delta vs 每周 sleep_score / hrv_avg），不做多变量回归——早期用户量不足，回归结果无意义。Gen 2 引入带 Wearable 协变量的混合效应模型。
早期数据不足规则：study_week < n（n = 待确认问题 #4 阈值）时，skin_sleep_correlation / lifestyle_skin_link 类洞察不触发计算，仅推送 milestone_approaching 或鼓励类提示，避免小样本 r 值误导用户。
expert_sources 【知识库元数据】专家知识文档注册表 — 追踪每条知识的来源、版本、授权状态；向量嵌入存向量数据库，此表存引用 source_id, expert_name (brenner / gunter / skin_specialist), content_type (paper / protocol / guideline / annotation), title, domain (hormone_skin / barrier_function / glycation / photoaging / elasticity / formulation), embedding_ref (向量数据库引用 ID), version, licensed_at, is_active (boolean) 知识库更新时手动录入；新论文/协议发布后由 Ruby 审核后加入。⚠ 向量化流程：expert_sources 新增记录后，后端触发一次嵌入任务，调用 embedding API（模型待定）将 content_type 对应文本转为向量，写入向量数据库，并将返回的向量 ID 回填到 expert_sources.embedding_ref——如果 embedding_ref = NULL，说明该条知识尚未入库，RAG 检索时不可用。licensed_at = Ruby 批准可用于产品的日期（非论文发表日），is_active = false 表示已下架（如版权争议）。
insight_rag_log 【RAG 审计】每条 insight_feed 生成时调用的知识源记录 — 合规审计 + 模型效果回溯 log_id, insight_id (FK → insight_feed), query_embedding_ref, source_ids_retrieved (array，来自 expert_sources), llm_model_version, prompt_version, generated_at 每次 insight_feed 生成时自动写入
sis_response_log 【闭环 Loop 1】用户看到 SIS 后的行为反馈 — 捕捉 SIS 作为干预依据的实际效果 user_id, session_id, study_week, sis_delta_shown, action_taken (viewed / changed_product / continued / paused / skipped), new_product_name (nullable，文本；Gen 1 无需 products 表，直接存名称), response_at 用户每周查看 SIS 后触发（App 内行为事件）。⚠ 被动查看的捕捉：App 在用户打开 SIS 结果页时自动写入一条 action_taken = viewed 的记录（无论用户是否有进一步操作），随后若用户点击"换产品/暂停"再覆盖更新 action_taken。这样可以区分"看了但不操作"（continued 行为的最保守估计）与"根本没看"（missing row）。工程师实现时，SIS 页面 onLoad 事件即触发写入，不依赖用户主动点击。
user_protocol_config 【闭环 Loop 4】基于 MCS 历史动态调整的采集协议配置 — 减少低质量采集 user_id, avg_mcs_score, consecutive_low_mcs_count, protocol_version, last_adapted_at 每周 SIS 计算完成后，后端从 sis_history.mcs_lowest（已为周级别最低值）聚合写入 user_protocol_config（UPSERT by user_id）。⚠ 正确聚合源是 sis_history.mcs_lowest，而非 images.mcs_level——前者已是每周两次采集的最低值，代表当周质量下限；若从 images 聚合需额外处理每张图像的 session/week 分组，反而容易出错。触发告警条件：consecutive_low_mcs_count 连续 ?周 = Low（周数待定）。⚠ 断链警告：当前架构缺少推送记录表——user_protocol_config 更新后 App 推送通知，但没有表记录"是否推送成功 / 用户是否点击"，Loop 4 的闭环效果无法量化。Gen 1 可暂时依赖 App 日志，但建议在 user_protocol_config 增加 last_push_sent_at / push_opened (boolean) 字段，或建独立 notification_log 表。
model_validation_log 【闭环 Loop 2】诊所 Ground Truth 与 AI 预测值的误差记录 — 触发模型再训练评估 user_id, session_id, visit_date, model_version, clinic_r2, clinic_r5, retraining_flagged (boolean), notes。Gen 1：仅记录诊所数据入库（clinic_records 新增）时的 model_version 与 retraining_flagged。无居家超声设备，无"居家 vs 诊所"误差比较逻辑——clinic_records 是唯一数据源，retraining_flagged 由 Ruby 人工评估后手动设置。Gen 2 建立图像→弹性预测模型后，扩展此表加入 predicted_r2/r5 与 delta_error 字段。 每次诊所数据入库（clinic_records 新增记录）后自动计算；误差超阈值则 retraining_flagged = true
cohort_insights 【闭环 Loop 3】跨用户聚合学习 — 按皮肤类型 × 干预类别分组的 SIS_delta 分布，输出推荐置信度 复合主键 (skin_type, fitzpatrick_scale, intervention_category)，skin_type, fitzpatrick_scale, intervention_category, sample_n, sis_delta_mean, sis_delta_p25, sis_delta_p75, confidence_level (high/medium/low), computed_at。⚠ intervention_category 必须与 intervention_log.category 使用同一套枚举；若两表枚举不同步，聚合结果无意义。⚠ 每月批量计算必须用 UPSERT（INSERT ... ON CONFLICT (skin_type, fitzpatrick_scale, intervention_category) DO UPDATE）而非 INSERT——否则每月新增一批重复行，表无限膨胀，insight_feed 引用的记录不确定。 每月批量计算（样本量 ≥ ? 才计入，阈值待统计顾问确认）。⚠ 下游写入路径：月度批处理任务完成 UPSERT 后，同一任务紧接着遍历所有 active 用户，用其 skin_type + fitzpatrick_scale 匹配 cohort_insights 记录，生成 insight_type = cohort_recommendation 的 insight_feed 行（注意：cohort_recommendation 的 session_id 可为 NULL，见 insight_feed 约束）。如果没有这一步，cohort_insights 表只是数据库里的数字，用户永远看不到"与你相同肤质的用户"洞察——批处理任务必须显式将结果写入 insight_feed，工程师不可遗漏。

工程师负责什么 vs. 你来决定什么

外包合同的核心边界。混淆这两列是项目失控最常见的原因。

工程师负责构建
  • App 采集流程（AR 定位引导、质检、MCS 逻辑、Context 表单、face_side 标记）
  • Wearable API 集成（Apple HealthKit / Oura / Dexcom）— OAuth 授权 + 每日定时同步
  • 图像上传 API — App → 云存储（S3）→ 触发 Haut.AI API 调用 → 写入 feature_vectors
  • 特征提取管线（AI 模型 API 集成）
  • 数据库 Schema 设计 + 增删改查
  • SIS 计算逻辑（算法由你定义，工程师实现）
  • 用户 Dashboard API — 返回 SIS、趋势图、特征历史
  • 诊所数据录入接口
  • 云端部署（AWS / GCP / 阿里云）
  • 数据加密、权限控制、合规基础设施
你来决定（科学 + 产品决策）
  • SIS 评分算法 — 哪些特征、什么权重、什么公式
  • MCS 阈值 — High / Medium / Low 的具体判断条件
  • 要采集哪些混淆因素字段 — Brenner 签字确认
  • AI 模型选型 — Gen 1 用哪个皮肤专项 API
  • 解剖学采集位点 — 哪几个面部区域、各几张
  • 干预分类体系 — 产品日志的品类定义
  • 诊所记录字段 — 哪些弹性测量指标必填
  • 哪侧是干预侧 — 默认左/右，还是用户自选
关键待决问题 — 工程启动前必须确认

Haut.AI 选哪些 biomarker 进入 SIS？（方向已定：手机拍摄 + Haut.AI；待确认具体特征权重）

这是整个处理层最核心的技术决策。工程师需要答案才能开始构建特征提取管线。

Gen 1 已选定：Haut.AI（手机照片 → 150+ biomarker 提取）。待皮肤科顾问确认：150+ biomarker 中选哪些特征进入 SIS 权重公式。Haut.AI 优势：支持手机自拍 + macro image + clinical image 统一 pipeline；300万+ 皮肤科标注图像训练；已有 clinically calibrated 纵向追踪。Gen 1 方向已定，无需评估其他选项。

选项 B — 通用视觉 API + 定制后处理（Google Vision、AWS Rekognition）：价格低、接入容易，但这些 API 输出的是通用标签（"面部"、"皮肤"、"花纹"），不是皮肤生物标志物。需要额外开发大量后处理层，工程量更高，科学质量更低。不能直接用来提取皮肤特征。

选项 C — 自训练模型：科学护城河最强。需要你还没有的标注皮肤镜训练数据。目标是 Gen 2 —— 等 Gen 1 积累了真实用户图像再训练。

建议：Gen 1 选 A（Haut.AI 或同类）。大规模积累真实用户图像。Gen 2 用这批数据训练自己的模型。数据集才是护城河，不是你起步用的模型。

合规

数据隐私与安全要求

月经周期、皮肤状况、睡眠等健康邻近数据有合规义务。这些要求必须从第一天开始内建，不能作为上线后的补丁。

🔒 存储安全
  • 静态加密 — 数据库 + S3 图像文件 AES-256 加密
  • 传输加密 — 所有 API 接口 TLS 1.2+
  • 图像访问控制 — S3 预签名 URL，禁止公开 Bucket
  • 数据库访问 — 不对外暴露，仅通过 API 层访问
  • Wearable OAuth Token — access_token / refresh_token 存密钥管理服务（AWS Secrets Manager），数据库只存引用 ID；Token 泄露等同于用户第三方账号暴露
👤 用户数据权利
  • 数据删除 — 用户可申请全量删除；工程师须实现跨表硬删除
  • 数据导出 — 用户可下载自己全部纵向数据
  • 同意记录 — 研究注册时的用户同意须带时间戳存储
  • 匿名化 — B2B / 研究用途的数据导出必须去除 PII
🏥 诊所数据访问
  • 角色权限控制 — 诊所员工只能上传自己患者的数据
  • 跨诊所隔离 — clinic_id 行级权限隔离
  • 操作日志 — 所有诊所上传记录带 staff_id + 时间戳
  • 用户授权 — 诊所记录关联须用户明确 opt-in
数据飞轮

四个闭环 — 从单向管道到自我强化系统

没有闭环的数据架构是数据湖，不是飞轮。以下四个回路让 DrRuby 的数据随时间自我改进，而不是静止积累。

研究层 · 轨道三

Brenner 研究组 Schema — 独立数据库，与消费者产品完全隔离

本 Schema 专供 Brenner 主导的 IRB 研究使用（N=1 个案研究 + 20-30人分组研究）。不与消费者数据库共享任何表——participant_id 与消费者 user_id 无直接关联，IRB 同意书版本独立追踪。工程师将其部署为独立 schema（研究环境），不进入产品主库。当前字段留白或标注 "?" 均属正常——待 Brenner 协议定稿后填入。

数据隔离原则：研究层数据库 ≠ 消费者数据库。研究参与者可同时是消费者用户，但两份数据通过匿名化 ID 隔离，DrRuby 产品工程师无权访问研究层原始数据。研究数据的控制权归 Brenner 所在研究机构，DrRuby 提供存储基础设施。

research_wearable_daily 【断层2修复】研究参与者 Wearable 日数据 — 独立于消费者 wearable_daily，通过三种路径写入 wearable_id (PK), participant_id (FK), session_id (FK), date, study_week, data_source (apple_health / oura / whoop / mira / manual_entry), sleep_score (nullable), deep_sleep_min (nullable), hrv_avg (nullable), resting_hr (nullable), skin_temp_delta (nullable), steps (nullable), lh_miu_ml (nullable，Mira 日测), e3g_ng_ml (nullable，Mira 日测), pdg_ng_ml (nullable，Mira 日测), data_entry_method (api_sync / manual / consumer_db_import)。⚠ consumer_db_import 路径：仅当 linked_consumer_user_id ≠ NULL 时，可从消费者 wearable_daily 拉取当日数据写入；大多数参与者保持 manual_entry 或 api_sync，确保研究数据独立性。 Mira 数据可在此表每日记录定量激素值，与 research_biomarkers 的周期性血检互补；研究层 Wearable 分析完全在此表进行，不依赖消费者库。⚠ UNIQUE 约束：(participant_id, date, data_source) 三字段联合唯一——防止 api_sync 和 manual_entry 同一天重复写入导致分析数据翻倍。
research_compliance_log 【断层3修复】每日依从性记录 — 参与者当天是否按协议执行干预，逐日追踪 compliance_id (PK), participant_id (FK), session_id (FK), intervention_id (FK → research_interventions), date, study_week, complied (boolean), dose_taken (nullable，实际剂量，若与协议不符时填写), non_compliance_reason (nullable，text), reported_at。⚠ research_interventions.compliance_rate 由此表聚合计算：COUNT(complied=true) / COUNT(*) by intervention_id，不可由参与者自报总体依从率替代。⚠ UNIQUE 约束：(participant_id, intervention_id, date) 三字段联合唯一——防止同一天重复打卡写入导致依从率计算错误。 App 每日推送打卡提醒（或参与者事后补填）；compliance_rate 低于 ?%（待 Brenner 定义）时，研究员收到告警
research_adverse_events 【断层4修复】不良事件实时记录 — 研究过程中任何不良反应即时上报，IRB 合规要求 event_id (PK), participant_id (FK), session_id (FK), event_date, study_week, event_description (text), severity (mild / moderate / severe / serious), related_to_intervention (boolean，nullable), intervention_id (nullable FK → research_interventions), reported_by (participant_self / researcher / dermatologist), reported_at, resolved_at (nullable), resolution_notes (nullable，text), irb_reported (boolean，是否已向 IRB 上报)。⚠ CHECK 约束：related_to_intervention = true 时 intervention_id 必须 NOT NULL——工程师必须在 DB 层加此约束，否则 IRB 审查时数据自相矛盾。。⚠ severity = serious 时，irb_reported 必须在 24 小时内置为 true；工程师需加自动提醒（邮件或系统通知）。 不可等 research_outcomes 填写时才记录；任何不良事件必须在发生时即时写入此表。研究结束时 research_outcomes.adverse_events 字段从本表聚合生成摘要
research_data_access_log 研究数据访问审计日志 — IRB 合规要求，记录谁在何时访问了哪位参与者的数据 log_id (PK), accessed_by (研究员ID), participant_id (FK), access_type (read / export / analysis), access_timestamp, access_reason, data_exported (boolean) IRB 通常要求数据访问可审计；所有访问必须有记录，不可删除
轨道三 · 工程部署说明
数据库隔离：研究 Schema 独立部署（独立 PostgreSQL database 或独立 schema with row-level security），消费者产品工程师无访问权限。研究数据不进入消费者分析 pipeline。

启动条件：research_protocol.status = 'approved' 前，research_participants 不可 INSERT 任何记录——IRB 号在提交阶段（irb_submitted）就会填写，但未批准前不可入组。工程师必须用 status 字段而非 irb_number IS NOT NULL 作为锁条件，否则 IRB 提交后批准前即可意外开始入组，违反伦理合规。
与消费者产品的数据共享（可选）：若参与者授权（linked_consumer_user_id 非 NULL），研究层的 research_skin_assessments 结果可作为参考写入消费者侧的 clinic_records，但仅限单向，消费者数据不回流进研究库。
